# AI-интеграция

## Что делает AI

Одним вызовом обращение анализируется по четырём осям:

| Ось | Значения | Назначение |
|-----|----------|------------|
| **sentiment** | positive / neutral / negative | тональность сообщения |
| **category** | job_offer / project / question / spam / other | тип обращения |
| **priority** | high / medium / low | приоритет ответа |
| **suggested_reply** | текст | черновик вежливого ответа |

Объединение трёх «функций» (анализ тональности + классификация + генерация
ответа) в **один вызов** экономит latency и токены — см.
[ADR-0003](adr/0003-ai-provider-abstraction.md).

## Провайдеры (паттерн Strategy)

Реализация выбирается настройкой `AI_PROVIDER`:

| Провайдер | Файл | Когда |
|-----------|------|-------|
| `mock` (по умолчанию) | `app/services/ai/mock_provider.py` | rule-based анализ, без сети и ключей |
| `anthropic` | `app/services/ai/anthropic_provider.py` | реальный Claude через tool-use |

Оба реализуют общий интерфейс `AIProvider.analyze()` из
`app/services/ai/base.py`, поэтому взаимозаменяемы.

### Mock / rule-based провайдер

Это **не пустая заглушка**, а детерминированный анализатор:
- тональность — по словарям позитивных/негативных слов;
- категория — по наборам ключевых слов (вакансия/оффер → `job_offer`,
  проект/заказ/ТЗ → `project`, спам-маркеры → `spam` и т.д.);
- приоритет — из категории;
- ответ — по шаблону под категорию.

Двойная роль: провайдер по умолчанию **и** fallback-слой (см. ниже). Благодаря
ему сервис запускается «из коробки» без ключей — важно для проверки тестового.

### Anthropic / Claude провайдер

Использует **forced tool-use**: модель обязана вызвать инструмент
`submit_analysis` с заданной JSON-схемой. Это гарантирует валидный
структурированный результат без парсинга «сырого» текста.

Включение:

```env
AI_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
ANTHROPIC_MODEL=claude-haiku-4-5-20251001
# ANTHROPIC_BASE_URL=https://your-proxy   # если нужен доступ из РФ
```

> Из РФ прямой доступ к Anthropic недоступен — нужен прокси-шлюз через
> `ANTHROPIC_BASE_URL`. Это осознанное ограничение, поэтому провайдером по
> умолчанию выбран `mock` (см. ADR-0003).

## Graceful fallback

Требование ТЗ: «если AI недоступен, сервис продолжает работать». Реализовано в
`app/services/ai_service.py` через **circuit breaker + fallback**:

```mermaid
flowchart TB
    start["analyze(name, comment)"] --> primary_mock{"primary == mock?"}
    primary_mock -->|"да"| mock["rule-based результат"]
    primary_mock -->|"нет"| breaker{"circuit breaker<br/>allow()?"}
    breaker -->|"разомкнут"| fb["fallback: rule-based"]
    breaker -->|"замкнут"| call["вызов Claude<br/>с таймаутом"]
    call -->|"успех"| ok["результат + record_success"]
    call -->|"ошибка / таймаут"| fail["record_failure → fallback: rule-based"]
```

- **Таймаут** (`AI_TIMEOUT_SECONDS`, по умолчанию 10с) — медленный AI не держит
  фоновую задачу.
- **Circuit breaker** — после 3 подряд сбоев «цепь размыкается» на 30с, и запросы
  сразу идут в fallback, не тратя время на мёртвый сервис. Через 30с — одна
  пробная попытка (half-open).
- **Fallback** — `MockAIProvider`, который не ходит в сеть и не падает.

Итог: обращение обрабатывается **всегда**, даже при полном отказе внешнего AI.
Флаг деградации виден в `GET /api/health` → `checks.ai_degraded`.

## Промпты

Вынесены в `app/services/ai/prompts.py` (версионируются вместе с кодом).

**System prompt** задаёт роль, перечисляет значения sentiment/category/priority
и правила составления черновика ответа (вежливо, 2–4 предложения, на русском, без
выдуманных фактов).

**Tool schema** (`submit_analysis`) фиксирует форму результата — `enum`-поля не
дают модели вернуть произвольные значения.

Полные тексты — в исходнике `prompts.py`.

## Что сделано с помощью AI (самоотчёт)

Проект написан в паре с AI-ассистентом (Claude, Anthropic). Прозрачно:

| Что генерировалось AI | Что правилось вручную |
|-----------------------|------------------------|
| Каркас слоёв, boilerplate (config, logging, session) | Уточнение слоёв, имена, доменные enum'ы |
| Rule-based словари и шаблоны mock-провайдера | Подбор ключевых слов под RU-домен разработчика |
| Схема circuit breaker | Логика half-open, разделение timeout/error |
| Тесты (структура, фикстуры) | Кейсы fallback, подмена sessionmaker для фона |
| Тексты промптов | Формулировки категорий под задачу лендинга |
| Docker/CI/compose | Нормализация DATABASE_URL, entrypoint-ретраи |

Правки после генерации: сортировка импортов и переход на `StrEnum` (ruff),
замена устаревшей константы `HTTP_422_*`, нормализация `postgres://`-URL под
async-драйвер, подмена фонового `sessionmaker` в тестах.
