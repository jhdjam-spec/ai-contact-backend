# ADR-0001: Вести Architecture Decision Records

## Status
Accepted
Date: 2026-07-22 · Deciders: @jhdjam-spec

## Context

Тестовое задание оценивает «как вы проектируете API и организуете архитектуру» и
«умение аргументировать решения». Устная аргументация теряется при передаче
проекта. Нужен письменный, версионируемый вместе с кодом след значимых решений.

## Decision Drivers

* Обязательно: прозрачность архитектурных решений для проверяющего.
* Желательно: минимальные накладные расходы на ведение.

## Considered Options

### Вариант 1: ADR в репозитории (MADR-формат)
- Плюсы: решения рядом с кодом, версионируются, стандартный формат.
- Минусы: нужно поддерживать дисциплину ведения.

### Вариант 2: Не документировать решения
- Плюсы: ноль усилий.
- Минусы: контекст решений теряется; проверяющий не видит аргументацию.

### Вариант 3: Wiki / внешний документ
- Плюсы: удобное редактирование.
- Минусы: отрывается от кода, рассинхронизируется, вне git-истории.

## Decision

Выбрали **Вариант 1**: ведём ADR в `docs/adr/` в формате MADR. Каждое значимое
решение — отдельный неизменяемый пронумерованный файл; индекс — в
`docs/adr/README.md`. Пересмотр решения = новый ADR со ссылкой на предыдущий.

## Rationale

ADR в репозитории — индустриальный стандарт фиксации «почему», решения живут в
git-истории и не рассинхронизируются с кодом. Прямо отвечает драйверу
прозрачности при минимальных усилиях.

## Consequences

- Positive: решения прозрачны, видна эволюция архитектуры.
- Negative: небольшие накладные расходы на ведение — оправданы для
  показательного проекта.
- Risks + Mitigation: ADR устаревают → статус (Accepted/Superseded) и ссылки
  между записями отражают актуальность.
