# ADR-0004: AI-анализ и email в фоновых задачах

## Status
Accepted
Date: 2026-07-22 · Deciders: @jhdjam-spec

## Context

Полный цикл обработки включает медленные операции: вызов AI (сотни мс — секунды)
и отправку двух писем по SMTP. Синхронное выполнение в обработчике запроса
заставит клиента ждать несколько секунд, а таймаут/сбой SMTP уронит весь запрос.

## Decision Drivers

* Обязательно: быстрый ответ клиенту, не зависящий от скорости AI/SMTP.
* Обязательно: сбой внешнего сервиса не ломает уже принятое обращение.
* Желательно: наблюдаемость статуса обработки.
* Желательно: минимум инфраструктуры (тестовое, один инстанс).

## Considered Options

### Вариант 1: Синхронно в обработчике запроса
- Плюсы: просто, результат AI сразу в ответе.
- Минусы: ответ ждёт AI+SMTP; сбой SMTP = ошибка запроса; плохой UX.

### Вариант 2: FastAPI BackgroundTasks
- Плюсы: ответ мгновенный (202); встроено во фреймворк; нет доп. инфраструктуры.
- Минусы: задачи живут в процессе приложения (не переживут рестарт, не шардятся).

### Вариант 3: Внешняя очередь (Celery / arq / RQ)
- Плюсы: устойчивость, ретраи, масштабирование на воркеры.
- Минусы: доп. брокер и воркер-процесс — избыточно для тестового (YAGNI).

## Decision

Выбрали **Вариант 2: BackgroundTasks**. Двухфазная обработка: (1) синхронно —
валидация, сохранение, ответ `202 Accepted` с id; (2) фоном — AI-анализ,
сохранение результата, рассылка писем, обновление статуса
(`received`→`processed`/`failed`). Фоновая задача создаёт собственную сессию БД.

## Rationale

BackgroundTasks закрывает оба обязательных драйвера (быстрый ответ, изоляция
сбоя) без внешней инфраструктуры. Внешняя очередь избыточна для одного инстанса и
объёма тестового; путь к ней зафиксирован как будущий шаг, а не реализован сейчас.

## Consequences

- Positive: ответ за десятки мс; сбой AI/SMTP не влияет на принятое обращение;
  статус-модель даёт наблюдаемость через `/api/metrics`.
- Negative: результат AI появляется после ответа (eventual consistency) —
  приемлемо для формы обратной связи; задачи не переживают рестарт процесса.
- Risks + Mitigation: рост нагрузки/многоинстансовость → вынести в arq/Celery
  (зафиксировано в docs/prr.md, раздел масштабирования).
