# ADR-0003: Абстракция AI-провайдера + graceful fallback

## Status
Accepted
Date: 2026-07-22 · Deciders: @jhdjam-spec

## Context

ТЗ требует минимум одну AI-функцию на backend с обязательным graceful fallback
(«если AI недоступен, сервис продолжает работать»). Ограничения:
- проверяющий должен запустить проект **без наших ключей** и увидеть работающий AI;
- из РФ прямой доступ к Anthropic/OpenAI недоступен (нужен прокси);
- fallback должен быть надёжным, а не декоративным.

## Decision Drivers

* Обязательно: graceful fallback — обращение обрабатывается при отказе AI.
* Обязательно: запуск «из коробки» без секретов у проверяющего.
* Обязательно: продемонстрировать реальную интеграцию с AI API.
* Желательно: лёгкая замена/добавление провайдера (GigaChat, OpenAI).
* Желательно: экономия latency и токенов.

## Considered Options

### Вариант 1: Только реальный AI (Anthropic), без абстракции
- Плюсы: меньше кода.
- Минусы: не запускается без ключа/прокси; нет fallback → нарушает ТЗ.

### Вариант 2: Абстракция провайдера + mock-заглушка (константы)
- Плюсы: запускается без ключа.
- Минусы: заглушка-константа выглядит как «AI не сделан»; fallback фиктивный.

### Вариант 3: Абстракция (Strategy) + rule-based провайдер как дефолт и fallback
- Плюсы: запуск без секретов; rule-based = осмысленный анализ И обязательный
  fallback-слой в одном; реальный Claude присутствует и документирован.
- Минусы: rule-based грубее LLM.

## Decision

Выбрали **Вариант 3**. Интерфейс `AIProvider.analyze()`; реализации
`MockAIProvider` (rule-based, дефолт) и `AnthropicAIProvider` (Claude, forced
tool-use). Выбор — флагом `AI_PROVIDER`. Надёжность в `AIService`: таймаут +
circuit breaker + fallback на mock. Один вызов возвращает все оси
(sentiment/category/priority/reply).

## Rationale

Rule-based провайдер закрывает сразу два драйвера: «запуск без секретов» и
«обязательный fallback» — без дублирования кода. Реальная интеграция Claude
показывает навык работы с AI API, но не обязывает проверяющего иметь ключ/прокси.
Circuit breaker бережёт время на заведомо мёртвом сервисе. Объединение осей в
один вызов экономит latency и токены.

## Consequences

- Positive: полное соответствие требованию fallback; нулевой порог запуска;
  добавление провайдера = новый класс без правок бизнес-логики.
- Negative: rule-based анализ грубее LLM (приемлемо как fallback и демо-дефолт).
- Risks + Mitigation: деградация AI незаметна → флаг `ai_degraded` в
  `/api/health`; сбои логируются структурно.
