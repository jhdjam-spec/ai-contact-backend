# Code Review Guideline

Как ревьюить изменения в этом проекте. Цель — держать код чистым, слоистым и
безопасным без бюрократии.

## Процесс

1. Изменения идут через Pull Request в `main`.
2. PR обязан проходить CI (ruff + tests + docker build) — красный CI не мержим.
3. Ревьюер проверяет по чек-листу ниже. Комментарии — конкретные, со ссылкой на
   `файл:строку`.
4. Merge — после approve и зелёного CI.

## Чек-лист ревьюера

### Архитектура и слои
- [ ] Изменение лежит в правильном слое (router → service → repository → model).
- [ ] Роутер не содержит бизнес-логики; репозиторий не течёт наружу (нет `select()`
      в сервисах).
- [ ] Нет циклических импортов между слоями.
- [ ] Новый внешний сервис заведён через абстракцию (Strategy), а не хардкодом.

### Корректность
- [ ] Есть тест на новую логику (happy path + хотя бы один негативный).
- [ ] Обработка ошибок явная; исключения не «глотаются» молча (кроме
      осознанного fallback с логом).
- [ ] Async-код не блокирует event loop (нет синхронного I/O в async-функциях).
- [ ] Сессии БД: request-scoped в запросе, отдельная — в фоне.

### Безопасность
- [ ] Пользовательский ввод валидируется и санитизируется (Pydantic, `extra="forbid"`).
- [ ] Нет секретов в коде/логах; чувствительные значения — из `.env`.
- [ ] Ошибки наружу не раскрывают внутренние детали (стектрейсы — только в лог).

### Наблюдаемость
- [ ] Значимые события логируются структурно (`logger.info(event, **fields)`),
      без f-строк с данными в ключе события.
- [ ] Логи не содержат секретов и лишних PII.

### Чистота
- [ ] `ruff check` проходит.
- [ ] Именование понятное; нет мёртвого кода и закомментированных блоков.
- [ ] Комментарии объясняют «почему», а не «что» (что видно из кода).
- [ ] Нет усложнения «на будущее» без потребности (YAGNI).

### Документация
- [ ] Значимое архитектурное решение оформлено как ADR.
- [ ] Изменение контракта API отражено в `docs/api.md` / OpenAPI.

## Что считается блокером

- Красный CI.
- Логика в неправильном слое.
- Непроваленный молча сбой (empty `except`).
- Секрет в коде или логах.
- Новый эндпоинт без валидации ввода или без теста.

## Автоматизация ревью (опционально)

Часть проверок делегирована инструментам, чтобы ревьюер смотрел на суть:
- **ruff** — стиль, импорты, распространённые баги.
- **pytest + coverage** — регрессии.
- **mypy** (конфиг в `pyproject.toml`) — типы; можно включить строгий прогон в CI.
