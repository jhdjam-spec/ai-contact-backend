## Чеклист по разделам

### 1) Product & Scope
Цель релиза — рабочий backend формы обратной связи с AI-обработкой (тестовое
задание). Входит: `POST /api/contact` (полный цикл), `/health`, `/metrics`, AI с
fallback, 2 письма, rate limit, логи, Swagger. Исключено (осознанно): фронтенд,
внешняя очередь, капча, нагрузочное тестирование.

### 2) Code & Architecture
Слоистая архитектура (api→services→repositories→models), внешние сервисы через
Strategy, DI через FastAPI. Долгов нет; известные ограничения зафиксированы в
ADR-0004 (BackgroundTasks вместо очереди). Критичных рисков в коде нет.

### 3) Quality & Testing
- Unit: rule-based анализ, circuit breaker, нормализация конфигурации.
- Integration: полный цикл contact→AI→email→metrics, health, OpenAPI.
- E2E smoke (живой uvicorn на SQLite, реальные HTTP через curl): **9/9 PASS** —
  health, contact (проект/вакансия/негатив), валидация (email/phone/extra→422),
  /docs, /openapi.json. Проверена фоновая обработка: 5 обращений → все
  `processed`, AI проставил sentiment/category/priority; создано 10 писем
  (5×2 владельцу+юзеру); лог `app.jsonl` содержит полную цепочку событий с
  request_id (`contact_received → contact_analyzed → emails_dispatched →
  contact_processed`).
- Результат последнего прогона: **29 passed, coverage 78%, ruff — clean**.
- Миграции: `alembic check` → «No new upgrade operations detected».

### 4) Security
Валидация Pydantic (+ `extra="forbid"`), санитизация control-chars, телефон→E.164.
Секреты вне репозитория (`.env` в `.gitignore`, `sync:false` в render.yaml).
Non-root контейнер. Ошибки не раскрывают внутренности (стектрейс — только в лог).
Слабое место для прода: `CORS_ORIGINS=*` (демо-дефолт) — сузить до домена.

### 5) Performance & Capacity
Async-стек; тяжёлое (AI/SMTP) вынесено в фон → ответ `202` за десятки мс.
Нагрузочные тесты не проводились (вне scope тестового). Bottleneck при росте —
in-process BackgroundTasks; путь решения известен (очередь).

### 6) Observability
Структурные JSON-логи в `logs/app.jsonl`, сквозной `request_id` (+заголовок
`X-Request-ID`), `GET /api/health` с реальной проверкой БД, `GET /api/metrics`.
Централизованный сбор/алертинг — вне scope (отмечено как post-release).

### 7) Deployment & Rollback
Выкат: push в `main` → GitHub Actions (lint→test→build→deploy) → Render Deploy
Hook. Миграции идемпотентны при старте. **Откат:** Render → Rollback на
предыдущий деплой (образ сохраняется), ~1-2 мин, ручной. Plan B при падении
prod-деплоя: сервис остаётся на предыдущем образе (Render не переключает трафик
на упавший healthcheck) → чиним и передеплой.

### 8) Operations & Support
README + docs/ (architecture, api, ai-integration, ci-cd, code-review) полны.
Runbook инцидентов и дежурство отсутствуют (не требуются для тестового).

### 9) Risks & Unknowns
